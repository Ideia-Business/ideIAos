# ADR — v14.4: Esquema de chave do COMPROVANTE assinado do step-up (verify-otp → agentd)

**Status:** **ACEITO (2026-06-24).** _Ratificado pelo operador — o esquema deste ADR foi **construído e provado por exit-code**: `source/agentd/stepup-verify-comprovante.mjs` (verify Ed25519/WebCrypto, autenticidade-ANTES-de-binding) passa nos proof-gates B3 (sig-inválida→3, não-pinada→4, hash-divergente→7, expirado→8), gate agregado 47/47 + mutação-testado, e **provado end-to-end contra o backend Supabase real** (F0b 2026-06-23: `verify-otp`→comprovante assinado→agentd verifica pubkey pinada exit 0; binding A≠B exit 7; single-use 400). O doc agora reflete a realidade implementada._ _(Histórico: PROPOSTO 2026-06-22 — fechou o gap de F0-build apontado pelo review adversarial de design, lente F0-readiness.)_ Pure design + cripto-local; não abre nenhum gate de mutação.
**Estende:** `v14.4-step-up-without-relying-party.md` (ACEITO/HYBRID) — aquele ADR cravou que `verify-otp` retorna um **comprovante ASSINADO** `{payload_hash, exp, jti}`, NUNCA um booleano (fix S-01). Este ADR especifica **com QUE chave** ele assina, **qual primitiva**, e **como o agentd verifica** — o que estava em prosa não-construível.
**Não revoga:** O2 (`v14.4-origin-auth-signing-mechanism.md`), o binding `payload_hash` (R-WP3), nem a fronteira loopback (S-08).
**Rules-piso:** `credential-isolation` (valor de chave nunca no contexto), `antifragile-gates` (prova por exit-code).

## Contexto — o gap

O pivô HYBRID (2026-06-22) introduziu um **artefato assinado REMOTAMENTE**: a edge function `verify-otp` (Supabase/Deno, plano P4) produz um comprovante que o **agentd-origem local consome e verifica ANTES de assinar o token O2**. Isso fecha o S-01 do precedente ideiapartner (que retornava `{verified:true}` booleano, forjável por qualquer coisa que alcance P1 com o `payload_hash` certo).

Mas nenhum documento dizia: (1) com **que chave** o `verify-otp` assina; (2) qual **algoritmo** (o O2 usa `ssh-keygen -Y` Ed25519 — uma primitiva LOCAL que a edge function Deno **não tem**); (3) **onde/como** o agentd pina e verifica a pubkey do backend P4. Sem isso, um implementador entrega um B3 que apenas **CASA** o `payload_hash` sem **VERIFICAR a assinatura** — reabrindo o confused-deputy que o HYBRID quer fechar. "Casar o hash" prova o *binding* (ação A ≠ ação B); **não** prova a *autenticidade* do comprovante.

## Decisão (proposta)

| Eixo | Decisão | Razão |
|------|---------|-------|
| **Quem assina** | Um **par Ed25519 DEDICADO do backend de step-up** (P4) — NÃO a `SERVICE_ROLE`, NÃO o JWT do Supabase. A **privada** vive só no env da edge function, referenciada **por NOME** (`STEPUP_SIGNING_KEY`), nunca no contexto/log/ledger. | `credential-isolation`: o valor nunca transita o LLM/contexto; chave de assinatura ≠ chave de acesso (least-privilege de posse) |
| **Primitiva** | **Ed25519 via WebCrypto** (`crypto.subtle`, nativo no Deno) sobre o JSON **canônico** do comprovante → assinatura detached (ou JWS compacto EdDSA). Declarada **DISTINTA** do `ssh-keygen -Y` do O2. | Deno não tem `ssh-keygen -Y`; nomear a primitiva concreta impede o verificador de confundir as duas cadeias de assinatura |
| **Payload do comprovante** | `{ payload_hash, sub, exp, jti, iat }` — `payload_hash` = sha256 do canonical do comando-alvo (R-WP3); `sub` = subject (email allowlisted); `jti` = nonce anti-replay; `exp` curto. | amarra o comprovante à AÇÃO específica e a uma janela curta |
| **Onde a pubkey é pinada** | A **pública** correspondente é **PINADA no agentd-origem no enrollment out-of-band** (mesma cerimônia/canal do pin O2/`enc_pubkey`). O schema de pin ganha a `stepup_pubkey` do backend P4 (pin **do backend**, um por backend — não por máquina). O pin **nunca** vem do payload/ref (sem downgrade/substituição). | a confiança é **autoritativa-LOCAL**, espelhando a FRONTEIRA-DE-PIN do O2; o ref/cloud nunca adiciona confiança |
| **Como o agentd verifica** | `stepup-token.sh` (origem) faz, **fail-closed**, nesta ordem: (1) **verifica a ASSINATURA** do comprovante contra a `stepup_pubkey` pinada — inválida → **exit≠0**; (2) confere `payload_hash` == hash do comando pretendido; (3) confere `exp` não-expirado; (4) confere `jti` não-replayed. **SÓ ENTÃO** assina o token O2. | autenticidade ANTES de binding; é a ordem que fecha o confused-deputy |

## Proof-gate (estende a condição #1 das 8 do step-up)

A condição #1 hoje prova "comprovante assinado-não-booleano + OTP da ação A não autoriza ação B" (binding). **Estende-se** para provar também **autenticidade**, por exit-code no bootstrap B3:

- Um comprovante com `payload_hash` correto mas **assinatura inválida** → **RECUSADO** (espelha `verify-payload.sh` **exit 3** = sig inválida).
- Um comprovante assinado por uma chave **NÃO-pinada** (mesmo que cripto-válida) → **RECUSADO** (espelha **exit 4** = chave não-pinada).
- Um comprovante válido mas `payload_hash` divergente do comando pretendido → **RECUSADO** (binding/R-WP3).
- Um comprovante com `jti` já visto → **RECUSADO** (anti-replay/R-WP8).

**Bootstrap autônomo (F0a, sem segredo real):** a prova roda LOCAL — gera-se um par Ed25519 de teste, pina-se a pública, assina-se um comprovante simulado, e prova-se por exit-code que sig-inválida / chave-não-pinada / hash-divergente / replay são recusados. **Não** exige o backend Supabase real (esse é F0b, ação humana gated). Verde aqui = o esquema de chave funciona, cripto-local.

## Consequências

**Positivas:** o "comprovante assinado" deixa de ser prosa e vira artefato construível e provável por exit-code; o confused-deputy (comprovante forjado que só carrega o hash certo) fica fechado por construção; reusa o padrão de pin + os exit-codes 3/4 que o `verify-payload.sh` do O2 já materializa (19/19).

**Aceitas (trade-offs):** o backend P4 ganha uma chave de assinatura própria (mais uma chave a custodiar — mitigado: privada só no env por nome, least-privilege, distinta da SERVICE_ROLE); o agentd ganha a `stepup_pubkey` no schema de pin (mais um item de enrollment out-of-band — aceitável, é o mesmo canal do O2).

**Fronteira:** este ADR especifica o **mecanismo de chave do comprovante**; não altera a topologia loopback (S-08), o tier×fator (O4 sempre no crítico/deploy), nem o que é gated. A revogação da `stepup_pubkey` é re-pin local out-of-band, como toda revogação autoritativa.
