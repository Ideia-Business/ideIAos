# ADR — v14.4: Step-up de aprovação humana (HYBRID: email-OTP universal + Touch ID local) (Q3)

**Status:** **ACEITO — REVISADO 2026-06-22 (pivô para HYBRID).** Originalmente aceito como
Touch-ID-via-agentd primário; **revisado no mesmo dia** após objeção do operador (Touch ID não é
universal na frota — Mac minis, teclados externos, futuros Linux/Windows não têm sensor) + consulta
ao **`@security-reviewer`** (veredito **ADOPT-WITH-CONDITIONS / HYBRID**). **Aceitar NÃO abre o gate:**
as 8 condições abaixo ainda exigem **prova por exit-code** no bootstrap (R-WP10). Reversível por edição.
**Resolve Q3** (mecanismo de step-up): email-OTP **ou** Touch ID, ambos amarrados à assinatura O2.
**Decisão irmã:** `docs/decisions/v14.4-origin-auth-signing-mechanism.md` (O2, assinatura por-máquina).

## Contexto

R-WP3 (contrato vivo) exige que a aprovação humana seja um **token de capability assinado, de uso
único**, com binding completo (`subject+role+action+ref+scope+nonce+expiry`) que **viaja com o
comando** — nunca uma sessão de browser. O problema (doc 70 §3): a aprovação humana acontece na
ORIGEM, mas o comando chega ao agentd-alvo como um **commit num ref**, não como sessão — então o
step-up tem que **produzir um artefato verificável que viaja com o comando**, não manter estado.

**A inversão de enquadramento (eixo da decisão):** "Touch ID vs email-OTP" é a pergunta errada.
Ambos são apenas **gates de presença-humana NA ORIGEM**. O que carrega a aprovação até a máquina B é
**a assinatura O2** (já provada em `source/agentd/{sign-payload,verify-payload}.sh`). O fator de
presença só decide *como o humano prova que está presente e aprova ESTA ação* na origem — depois,
o agentd assina o token. Logo o backend do email-OTP tem blast-radius muito menor do que parece:
ele é o **sensor de presença na origem**, nunca o transporte cross-máquina.

**Objeção que motivou o pivô:** Touch ID (`LocalAuthentication`) é **offline e não-phishável**, mas
**não-universal** — metade da frota não tem sensor. Apoiar o 2º fator em hardware ausente é frágil.

## Decisão (HYBRID)

**Fator de presença na origem = email-OTP como mecanismo PRIMÁRIO e UNIVERSAL, com Touch ID como
ATALHO LOCAL opcional onde o hardware existir.** Ambos convergem no mesmo ponto: o agentd-origem
assina o token de capability O2. O contrato R-WP3 **não muda** (é agnóstico ao mecanismo).

| Eixo | Decisão | Razão |
|------|---------|-------|
| **Fator primário (universal)** | **email-OTP** a `gustavo@` (validado), via **projeto Supabase DEDICADO** `ideiaos-cockpit-stepup` | funciona em qualquer host com browser+email; resolve a não-universalidade do Touch ID |
| **Atalho local (onde houver)** | **Touch ID** (`LocalAuthentication`-via-agentd) — offline, sem rede | preserva a propriedade "zero rede" do design original nos hosts capazes; não desperdiça a biometria dos MacBooks |
| **Binding obrigatório (o fix crítico S-01/S-03)** | o OTP é amarrado a `payload_hash = sha256(BYTES do command-file producer-canonical)` — **o produtor canonicaliza**, e o MESMO hash viaja client→send-otp→comprovante (`stepup-token.sh:54` = `shasum -a 256` dos bytes). O field-set **mínimo same-machine** (regime F0a provado) é `{action,ref,scope,sub,nonce}` (target = self); o command-file **cross-máquina** (Wave gated) DEVE adicionalmente incluir **`target_machine`** — senão um OTP/comprovante colhido p/ máquina A autorizaria a MESMA ação em B (confused-deputy cross-máquina). `verify-otp` retorna um **comprovante assinado do binding** (`{payload_hash,sub,exp,jti,iat}`), **NUNCA um booleano** | sem isso, um OTP colhido p/ "rotate VERCEL" autorizaria "revoke SERVICE_ROLE" (confused-deputy/replay). O agentd só assina o token O2 se o comprovante casar o `payload_hash` do comando |
| **O artefato cross-máquina** | token O2 `{subject,role,action,ref,scope,nonce,expiry,otp_proof}` assinado pela chave de máquina | não é sessão; é a prova que viaja no ref e o agentd-alvo verifica (`verify-payload.sh`) |
| **Verificação no alvo** | rejeita token expirado, `nonce` reusado (registro durável), binding divergente | anti-replay (R-WP3 + Q4); fail-closed |
| **Backend = projeto DEDICADO** | `ideiaos-cockpit-stepup`: só send-otp/verify-otp + tabelas otp/trusted-device, SERVICE_ROLE isolada, ZERO dado de produto, CORS só loopback | blast-radius se comprometido = "forjar aprovações", **não** "vazar dados de produto" (decisão do operador 2026-06-22) |

### Matriz tier × fator

| Tier | Exemplo | Fator de presença | "Lembrar deste PC" pula OTP? | O4 out-of-band |
|------|---------|-------------------|------------------------------|----------------|
| local-reversível (v14.1) | pause autosync | nenhum (armar/disparar) | n/a | não |
| **sensível** | rotate cred sensível, same-machine | OTP **ou** Touch ID | ✅ só same-machine, **≤7d** | não |
| **alto** | rotate cred alto, cross-máquina | OTP/Touch ID **toda vez** | ❌ | não |
| **crítico** | rotate/revoke SERVICE_ROLE | OTP/Touch ID **toda vez** | ❌ | ✅ **sempre** |
| **deploy** | deploy de produção | OTP/Touch ID **toda vez** | ❌ | ✅ **sempre** |

## Alternativas consideradas (e por que não)

- **Touch-ID-only (a decisão original deste ADR)** — rejeitada: **não-universal** (Mac minis/teclados externos/Linux/Windows sem sensor). Mantido só como atalho local opcional.
- **WebAuthn/passkey no browser** — rejeitada: exige relying-party server; passkey em loopback sem RP é não-trivial.
- **Reusar o projeto Supabase do produto (ideiapartner)** — rejeitada (`@security-reviewer` S-04): acoplaria o step-up de mutação de produção de TODA a constelação à superfície de um app de produto (mesma SERVICE_ROLE/migrations). Projeto dedicado isola o blast-radius.
- **verify-otp retornando booleano (como no ideiapartner as-is)** — rejeitada (S-01): viola R-WP3 (binding); abre replay/confused-deputy. Exige o comprovante assinado do `payload_hash`.

## Consequências

**Positivas:** universal na frota (qualquer host com email); reusa padrões production-grade do
ideiapartner (RLS deny-all, lockout, rate-limit); o backend é só o sensor de presença na origem;
HYBRID preserva o caminho offline (Touch ID) onde existe.

**Aceitas (custo honesto — `operating-discipline` §6):**
- **Dependência de backend reintroduzida:** o step-up depende de Supabase + email + rede — o que o
  local-first havia eliminado. Pago pela universalidade de frota. **Mitigado:** fail-CLOSED (backend
  fora → recusa assinar, nunca pula) + projeto dedicado + O4 out-of-band no crítico (Supabase **não**
  é SPOF do tier crítico).
- **Email-OTP é mais fraco que Touch ID:** prova "controle do email", não "presença biométrica" —
  phishável e vulnerável a comprometimento da conta de email (vetores que o Touch ID offline não
  tinha). Mitigado: binding (não-replay), O4 no crítico (email-comprometido sozinho não rotaciona
  SERVICE_ROLE), remember-device só `sensível` same-machine.
- **Q4 residual (relógio/replay):** registro de nonces-vistos durável + tolerância de clock — detalhe
  de implementação a calibrar.

**As 8 condições de adoção (BLOQUEANTES — verificadas por exit-code no bootstrap):**
1. **Binding obrigatório** (S-01/S-03): `verify-otp` retorna comprovante assinado do `payload_hash`, nunca booleano. Provar: OTP da ação A **não** autoriza ação B.
2. **Projeto Supabase DEDICADO** (S-04): `ideiaos-cockpit-stepup`, SERVICE_ROLE isolada, zero dado de produto.
3. **Fail-CLOSED** (S-02): backend/email/rede fora → agentd recusa assinar (exit≠0), nunca degrada para skip.
4. **OTP como fator efêmero sensível** (credential-isolation): OTP nunca em log/snapshot/ledger/contexto-LLM; só a assinatura do binding viaja.
5. **Tiering de remember-device** (S-05): skip de OTP só `sensível` same-machine, janela ≤7d, token bind a `max_tier=sensível`. `alto`+ sempre OTP; `crítico`/`deploy` sempre O4.
6. **CSPRNG** (S-06): `crypto.getRandomValues`, não `Math.random` (o gerador do ideiapartner não serve aqui).
7. **Rate-limit ancorado em email/subject, não IP** (S-07) + **CORS só loopback** (S-08).
8. **Sem `signInWithPassword`** de produto (S-09); Touch ID continua o caminho offline preferencial onde existir.

## Reusabilidade do ideiapartner (minerar adaptado, não copiar)

- **Reusar o padrão:** RLS deny-all de `otp_codes`, funções `is_otp_locked_out`/`check_otp_rate_limit`, sanitização de input, padrão CORS+`sanitizeErrorForClient`.
- **Adaptar:** `send-otp`/`verify-otp` (+`payload_hash` binding, CSPRNG, rate-limit não-IP, comprovante assinado); remember-device (tiering).
- **NÃO copiar:** `signInWithPassword`, roles de produto (`user_roles`/`is_seller`), `admin_emails`, domínios de produto na CORS.
