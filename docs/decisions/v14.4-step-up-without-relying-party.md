# ADR — v14.4: Step-up de aprovação humana sem relying-party server (Q3)

**Status:** **PROPOSTO** (2026-06-22) — recomendação fundamentada; **aguarda aceitação do operador** e prova de viabilidade por exit-code.
**Propõe a resolução de Q3** (design do step-up sem RP-server) do precursor `docs/ideiaos-console/70-security-v14_4-threat-model-precursor.md` §3/§8 — o **mecanismo primário** (`LocalAuthentication` de daemon) é incerteza-chave, **condicionado à prova de viabilidade por exit-code**; o fallback de terminal garante o design mesmo se o primário for inviável (ver Consequências).
**Pré-condição do gate R-WP10** (`specs/cockpit/spec.md`): para ações que exigem step-up (tiers `sensível`/`alto`/`crítico`), o write-path não habilita até este ADR ser **Aceito** E o caminho de step-up ser **provado viável por exit-code**.
**Decisão irmã:** `docs/decisions/v14.4-origin-auth-signing-mechanism.md` (O2, assinatura por-máquina).

## Contexto

R-WP3 (contrato vivo) exige que a aprovação humana seja um **token de capability assinado, de uso
único**, com binding completo (`subject+role+action+ref+scope+nonce+expiry`) que **viaja com o
comando** — nunca uma sessão de browser — e que o segredo de assinatura **não** resida no browser.

O problema (doc 70 §3): **WebAuthn/passkey assina um desafio de um relying-party server** — e a
arquitetura **nega backend** (ADR `v14-cockpit-local-first-git-as-bus`). Sem RP-server, uma passkey
de browser em loopback é não-trivial. E a sessão do browser-origem **nunca alcança** o agentd-alvo
(o comando chega como um commit no ref, não como requisição HTTP autenticada).

## Decisão (proposta)

Adotar **step-up via `LocalAuthentication` (Touch ID) colhido pelo `ideiaos-agentd`-origem
server-side, não pelo browser**. O browser apenas **dispara** o pedido; o agentd-origem **colhe** o
Touch ID e **assina** o token de capability com a chave de máquina (O2 do ADR irmão).

| Eixo | Decisão proposta | Razão |
|------|------------------|-------|
| **Onde a aprovação acontece** | no **agentd-origem** (processo local, fora do browser), via `LocalAuthentication` do macOS | mantém o segredo de assinatura fora do browser (R-WP3/`credential-isolation`); não precisa de RP-server |
| **O artefato** | **token de capability de uso único**: `subject+role+action+ref+scope+nonce+expiry`, assinado pela chave de máquina (O2), encapsulado no comando | não é sessão; é prova verificável que viaja com o comando até o alvo (R-WP3) |
| **Verificação no alvo** | o agentd-alvo rejeita token expirado, `nonce` reusado (registro de nonces-vistos), ou binding que não casa com o comando | anti-replay (R-WP3 + Q4); fail-closed |
| **Tier `critical`/`deploy`** | **+ O4 out-of-band** (confirmação fora do git) **além** do step-up | defesa-em-profundidade mesmo se o step-up/assinatura falhar (doc 70 §4/§6) |
| **Fallback se LocalAuthentication-via-daemon não for viável** | confirmação **explícita no terminal da máquina-origem** (ato humano local), ainda assinada pela chave de máquina; sob fallback, o **O4 out-of-band torna-se obrigatório para `sensível`+** | **garantia reduzida**: prova presença-no-host, NÃO presença-de-humano-biométrico (num host comprometido, um processo residente satisfaz o terminal sem humano). Modo degradado aceitável; o O4 compensa a perda do fator biométrico |

## Alternativas consideradas (e por que não)

- **WebAuthn/passkey no browser** — rejeitada: exige relying-party server; a arquitetura nega backend. Passkey em loopback sem RP é não-trivial e arrastaria infra que o local-first evita.
- **Manter sessão do browser-origem até o alvo** — impossível: o comando chega como commit no ref; a sessão HTTP não atravessa máquinas. Por isso o step-up tem que **produzir um artefato** (token), não manter estado.
- **Sem step-up (só assinatura de máquina)** — insuficiente para mutação de produção: a assinatura prova *qual máquina*, não *que um humano aprovou aquela ação específica*. R-WP3 exige a prova de aprovação humana para os tiers sensível+.

## Consequências

**Positivas:** o "touch" humano vira artefato verificável de uso único sem backend; segredo de
assinatura fora do browser; reusa a chave de máquina (O2); anti-replay explícito; O4 cobre o tier
crítico independentemente.

**Aceitas (riscos residuais declarados):**
- **Q3 residual (viabilidade):** `LocalAuthentication` a partir de um contexto de **daemon**
  (LaunchAgent) pode ter restrições de entitlement/UI no macOS. **Esta é a incerteza-chave** — por
  isso o ADR exige **prova por exit-code** antes da habilitação; se inviável, cai no fallback de
  confirmação-no-terminal (sem bloquear o resto do design).
- **Q4 residual (relógio/replay):** o registro de nonces-vistos no alvo precisa de estado durável;
  o `expiry` exige relógios minimamente sincronizados entre máquinas assimétricas. Tratado em R-WP3
  (cenário de replay) — tolerância de clock é detalhe de implementação a calibrar.

**Pré-condições de bootstrap (BLOQUEIAM a habilitação, verificáveis por exit-code):**
1. Provar que o agentd colhe Touch ID via `LocalAuthentication` e produz um token assinado — **ou**
   confirmar o fallback de terminal.
2. Provar a verificação fim-a-fim do token no alvo: aceita token válido/inédito; **rejeita**
   expirado, `nonce` reusado, binding divergente (R-WP3 fail-closed) — por exit-code.
