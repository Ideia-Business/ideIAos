## O que é o gsd-browser
Descrever: CLI Rust com daemon persistente, 92 comandos via CDP, daemon auto-start, sessões nomeadas, versioned refs (@v1:e1), assertions determinísticas (assert + batch), visual-diff nativo, auth vault com credenciais criptografadas, output JSON estruturado. ## Skill Piloto Avaliada
Referência direta a source/skills/frontend-visual-loop/SKILL.md.
Motor atual: mcp__chrome-devtools__* (take_snapshot, take_screenshot, resize_page, list_console_messages, lighthouse_audit).
Papel hipotético do gsd-browser nesta skill:
- gsd-browser navigate + snapshot + click-ref → substitui navigate_page + take_snapshot
- gsd-browser screenshot → substitui take_screenshot
- gsd-browser assert → adiciona verificação determinística que o chrome-devtools MCP não tem nativamente

- gsd-browser vault-login → relevante para fluxos atrás de login (upgrade já mencionado na skill)

## Análise por Critério

### 1. Custo de token
chrome-devtools MCP: cada chamada MCP tem overhead de round-trip + schema de ferramenta na janela. A11y tree via take_snapshot produz 2-5KB de output semântico por página.
gsd-browser: é um comando CLI — o executor chama Bash, não uma ferramenta MCP. O output (--json) é compacto e estruturado. Nenhuma ferramenta adicional entra no contexto visível do modelo.
Vantagem potencial: eliminar o overhead de definição de ferramentas MCP do chrome-devtools da janela de contexto (estimativa: ~500-1500 tokens por sessão com o MCP ativo).
Incerteza: sem medição real no contexto IdeiaOS; a vantagem é estrutural, não medida.

### 2. Determinismo
chrome-devtools MCP: seletores frágeis, sem sistema de refs versionadas. Interações dependem de timing e estado implícito da página.
gsd-browser: versioned refs (@v1:e1) — cada snapshot congela os refs; re-snapshot gera nova versão. assert com JSON de checks. batch para sequências atômicas. Mecanismo de determinismo explícito e documentado.
Vantagem clara: gsd-browser é arquiteturalmente mais determinístico para automação de UI.

### 3. Esforço de instalação (binário Rust)
Sem Rust toolchain necessária (binário pré-compilado). Instalação: curl -fsSL https://install.gsd.build/browser | bash — script que baixa o binário darwin-arm64.
Riscos: (a) curl | bash é um vetor de risco de supply chain — deve ser revisado antes de qualquer execução real; (b) npm e crates.io não publicados ainda — depende do GitHub Releases do org gsd-build; (c) versão não fixada no installer.
Mitigação sugerida: baixar binário diretamente do release tag (não via curl | bash) + verificar hash antes de execução.

### 4. Risco de churn de org
Org: gsd-build (github.com/gsd-build/gsd-browser). Distinto de: open-gsd (GSD Redux — o que o IdeiaOS usa), gsd-pi (outra linha).
Atenção: gsd-build é o org do gsd-browser; não confundir com open-gsd (nosso GSD). São organizações separadas no GitHub.
Pacotes npm e crates.io não publicados ainda → dependência do binário do GitHub Releases = ponto único de falha se o org mover/arquivar o repo.
Risco: médio. Ferramenta nova (sem histórico longo de manutenção verificável), org desconhecida além deste repo.

## O que é o agent-inbox
MCP server que cria endereços de e-mail temporários reais sob demanda. Sem API keys. Sem contas. Providers: mail.tm (primário) com fallback para 1secmail. Inboxes em memória — não sobrevivem a restart do servidor. Instalação via npx: "npx -y gsd-agent-inbox".
6 ferramentas: create_inbox, check_inbox, wait_for_email, verify_email, list_inboxes, delete_inbox.
Inboxes nomeados: create_inbox com name: "main" → referenciável por nome em chamadas subsequentes.
Auto-cleanup: inboxes deletados no SIGINT/SIGTERM do servidor.

## Quando usar (opt-in)
- Testes de fluxo de sign-up com verificação de e-mail (Supabase Auth, Resend, SendGrid, AWS SES)
- Testes de convite por e-mail / magic link
- Qualquer cenário de teste onde um agente precisa receber e-mail sem conta real

## Quando NÃO usar (proibições absolutas)
- NUNCA em ambiente de produção
- NUNCA com inboxes persistidos (dados sensíveis não sobrevivem restart de qualquer forma — mas não tentar persistir)
- NUNCA para serviços que bloqueiam domínios descartáveis sem alertar o usuário primeiro
- NUNCA instalar como MCP global sem consentimento explícito (@devops autoriza)

## Higiene de MCP (referência a source/rules/common/mcp-hygiene.md)
Classificação de risco: Médio (acesso a rede externa — mail.tm e 1secmail).
Regra de higiene: habilitar somente durante a sessão de teste; desabilitar após.
Limite: não contar como um dos ≤10 MCPs ativos permanentes.
O agent-inbox NÃO deve aparecer em ~/.claude/settings.json como MCP permanente — usar configuração de projeto (`.claude/settings.json` do projeto-filho com disabledMcpServers quando não em uso ativo).

## Como ativar (por sessão, exclusivo @devops)
Ativação manual por projeto-filho (nfideia, ideiapartner, cfoai-grupori):

Opção A — Claude Code (temporária, por sessão):
  claude mcp add agent-inbox -- npx -y gsd-agent-inbox

Opção B — Configuração de projeto (.claude/settings.json do projeto-filho):
  Adicionar em mcpServers: { "agent-inbox": { "command": "npx", "args": ["-y", "gsd-agent-inbox"] } }
  Após o teste: mover para disabledMcpServers ou remover.

Não usar o auto-installer interativo (npx gsd-agent-inbox em TTY) — ele detecta o Claude Code instalado e modifica ~/.claude/settings.json globalmente, violando a regra de opt-in.

## Como usar (padrão de chamada)
Sequência típica para verificação de e-mail em teste:
1. create_inbox com prefix e name descritivos (ex.: prefix: "signup-test", name: "test-main")
2. Executar o fluxo de sign-up no produto usando o endereço retornado
3. verify_email (endereço: "test-main", subject_contains: "confirm") — one-shot: aguarda, extrai link, acessa via HTTP
4. Verificar estado esperado no produto
5. delete_inbox (endereço: "test-main") — limpar imediatamente após o teste

## Como desativar após uso
  claude mcp remove agent-inbox
  (ou: remover do .claude/settings.json e reiniciar sessão)

## Limitações conhecidas
- Alguns serviços bloqueiam domínios descartáveis (mail.tm, 1secmail) — se sign-up for rejeitado, reportar ao usuário; não tentar contornar
- Inboxes não sobrevivem restart do MCP server — projetar testes como stateless
- Sem suporte a anexos — text/HTML apenas
- Provider fallback automático (mail.tm → 1secmail): sem controle sobre qual domínio será usado

## Skill do agent-inbox (opcional)
O agent-inbox distribui uma SKILL.md que ensina o agente a usar o inbox automaticamente em fluxos de auth. Instalação manual se desejado:
  mkdir -p ~/.claude/skills/agent-inbox
  curl -fsSL https://raw.githubusercontent.com/gsd-build/agent-inbox/main/skill/SKILL.md -o ~/.claude/skills/agent-inbox/SKILL.md
Instalação da skill é operação de @devops e requer revisão do conteúdo antes de executar o curl.
