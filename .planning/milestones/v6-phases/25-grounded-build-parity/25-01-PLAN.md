ESTRUTURA DO SKILL (baseada em deep-research/SKILL.md como referência de formato):

## Quando usar
- Criar um novo agent (`source/agents/`) ou nova skill (`source/skills/`) cujo domínio não está mapeado na codebase.
- Quando spec genérica produziria persona sem fundamento real (anti-pattern).

## Quando NÃO usar
- Agent/skill cujo domínio já está documentado em docs/research/ com fontes recentes (< 90 dias).
- Ajustes incrementais em agent existente — use o agent diretamente.

## Processo (4 fases, inspirado no Design Phase do OpenSquad)

### Fase 1 — Definir domínio e tipo
Coletar do usuário: (a) domínio-alvo (ex.: "copywriting para LinkedIn"), (b) tipo de artefato (agent ou skill), (c) contexto de uso (quando será invocado, quem chama).

### Fase 2 — Pesquisa fundamentada (delegar a /deep-research)
Delegar pesquisa iterativa (máx 3 ciclos) à skill /deep-research com as seguintes queries obrigatórias para o domínio recebido:
1. "{domínio} frameworks and best practices" — extrair: 1-2 frameworks operacionais com etapas concretas.
2. "{domínio} mistakes to avoid anti-patterns" — extrair: ≥4 "nunca faça" com explicação de por quê prejudica.
3. "{domínio} quality criteria evaluation rubric" — extrair: critérios mensuráveis de sucesso.
4. (Opcional, se domínio envolve escolha de modelo/tool) "{task type} LLM model selection" — extrair: quando usar opus vs sonnet vs haiku para este tipo de tarefa.

Para cada achado: registrar fonte (URL + data). Sem fonte = não entra na spec.

### Fase 3 — Model routing com justificativa
Com base nos achados de pesquisa, justificar a escolha de modelo:
- opus: raciocínio estrutural de alto impacto, planejamento, auditoria de segurança.
- sonnet: implementação, review, análise de código — equilíbrio custo/qualidade.
- haiku: tarefas leves, observação, metadata — máxima velocidade e mínimo custo.
Registrar o racional na spec, não apenas o nome do modelo.

### Fase 4 — Produzir spec grounded
Produzir o arquivo de destino (`source/agents/<nome>.md` ou `source/skills/<nome>/SKILL.md`) com:
- Frontmatter: name, description (agents: + model, tools).
- `# SOURCE: IdeiaOS v2` logo após frontmatter.
- Seções: Quando usar / Quando NÃO usar / Processo (derivado dos frameworks pesquisados) / Anti-patterns (derivados da pesquisa, com explicação) / Qualidade (critérios mensuráveis da pesquisa).
- Bloco `## Fontes` ao final: lista de URLs + datas usados na pesquisa. Mínimo 2 fontes externas. Sem fontes = spec não é grounded e deve ser rejeitada.

## Output esperado
Arquivo `source/agents/<nome>.md` ou `source/skills/<nome>/SKILL.md` com:
- Seção `## Fontes` com ≥2 entradas externas citadas.
- Anti-patterns com justificativa (não lista genérica).
- Model routing com racional documentado (para agents).

## Anti-patterns
- Gerar spec sem executar a Fase 2 (pesquisa) — produz persona genérica.
- Citar fontes sem URL + data — não auditável.
- Copiar frameworks sem adaptar ao padrão IdeiaOS (frontmatter, SOURCE, idioma pt-BR).
- Reutilizar anti-patterns genéricos sem derivá-los de pesquisa do domínio específico.

## Relações
- Delega pesquisa a `/deep-research` (iterative retrieval).
- Produz artefato compatível com `source/agents/` ou `source/skills/` que passa em `validate_agent_contracts` (se agent).
- Resultado pode ser inspecionado por `/ideiaos-catalog`.

Adicionar a flag `--validate-parity` ao `build-adapters.sh` existente. Bash 3.2 estrito, sem jq.

PARSING da flag no bloco while existente (após `--dry-run`):
```
--validate-parity) VALIDATE_PARITY=true; shift ;;
```
Declarar `VALIDATE_PARITY=false` junto com as demais variáveis no topo.

Atualizar `usage()` para incluir:
```
  --validate-parity: check semantic equivalence between claude and cursor targets
```

IMPLEMENTAR função `validate_parity()` logo após `validate_agent_contracts()`.

Lógica da função (bash 3.2, sem jq, leitura linha a linha de modules.json):

```
validate_parity():
  1. Para cada módulo em modules.json com "kind": "skill" ou "kind": "agent":
     - Extrair id, targets[], source
     - Determinar se está em claude-targets e/ou cursor-targets

  2. Para cada par (claude-only vs cursor-only):
     - Se módulo aparece em targets do claude mas não cursor: registrar divergência CLAUDE_ONLY
     - Se módulo aparece em targets do cursor mas não claude: registrar divergência CURSOR_ONLY
     - Divergência só é ERRO se o módulo NÃO tem "installStrategy": "manual"
       (manual = instalação intencional seletiva = justificativa implícita)

  3. Ao final: se divergências_erro > 0 → imprimir relatório + exit 1.
```

PARSER de modules.json sem jq (bash puro): ler o arquivo linha a linha coletando estado.
Usar variáveis de estado: dentro de `{`, acumular `current_id`, `current_targets_claude`, `current_targets_cursor`, `current_strategy`, `current_kind`. Ao encontrar `}` ou `},`, avaliar e resetar.

Estratégia de parsing:
- `"id":` → extrair valor entre aspas após `:`
- `"targets":` → iniciar coleta de targets até `]`
- `"claude"` dentro de targets → setar flag has_claude=1
- `"cursor"` dentro de targets → setar flag has_cursor=1
- `"installStrategy":` → extrair valor (ex: "manual", "always", "stack:react")
- `"kind":` → extrair valor (skill, agent, hook, template, etc.)
- Fechar bloco `}` → avaliar apenas se kind == skill ou kind == agent

RELATÓRIO de saída (impressão linha a linha em >&2 se houver erros):
```
→ Validating cross-target parity...
  DIVERGENCE: skill-banner-design — in claude+cursor but NOT cursor  [exemplo de erro]
  ...
ERROR: N parity divergence(s) found. Add to both targets or set installStrategy: manual.
```
Se tudo OK:
```
✓ Cross-target parity OK (N modules checked)
```

validate_agent_contracts
if $VALIDATE_PARITY; then validate_parity; fi

  ...
