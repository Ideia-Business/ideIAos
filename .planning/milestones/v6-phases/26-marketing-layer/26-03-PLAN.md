Header `# SOURCE: OpenSquad MIT renatoasse/opensquad | adapted: IdeiaOS v6` na primeira linha do corpo (markdown `#`, nunca `<!--`).

A skill conduz o pipeline (R6-05), adaptado do OpenSquad ao idioma IdeiaOS:
1. **Discovery** — recebe o pedido em linguagem natural; classifica formato(s)-alvo; pergunta o minimo (publico, tom, plataforma) — max ~3 perguntas, uma por vez (padrao discovery.prompt.md). Oferece investigacao opcional via skill `marketing-research` (Plano 02). NAO recriar o wizard de 8 perguntas inteiro — versao enxuta IdeiaOS.
2. **Design** — consulta a best-practice do formato em `source/rules/marketing/{formato}.md` (indice em source/rules/marketing/README.md) + a disciplina relevante (copywriting/strategist). Recruta o mkt-estrategista (opus) para angulos + big idea; apresenta angulos ao usuario (checkpoint).
3. **Build** — mkt-copywriter (sonnet) produz 3 hooks -> body -> CTA no formato escolhido (best-practice injetada em runtime, como o "format injection" do runner.pipeline.md). Checkpoint de aprovacao de conteudo ANTES de qualquer passo visual (regra Content Approval do design.prompt.md). Depois mkt-designer (sonnet) gera a peca visual REUSANDO a suite de design (banner-design/slides/ui-ux-pro-max).
4. **Review** — mkt-revisor (sonnet) faz scoring + veto contra review.md; rejeicao volta ao copywriter com feedback (max 2 ciclos — loop on_reject do runner).
5. **Publish** — OPCIONAL/MANUAL. As skills de publicacao do OpenSquad (blotato, resend, instagram-publisher) sao MCP/API-key dependentes -> marcar como passo manual/deferido: o /marketing entrega os arquivos finais e instrui a publicacao manual ou via skill MCP futura. NAO implementar publicacao automatica neste plano.

Gates binarios: entre fases, validar saida com `test -s` (padrao R6-01/OpenSquad) antes de avancar — nao confiar no Read tool. Documentar transparencia ao estilo Deia ("mostro o que vou fazer antes de fazer").

Saida do trabalho: gravar pecas em um diretorio de output do projeto-alvo (ex.: `docs/marketing/{data}-{slug}/`) — definir convencao no SKILL.md. NAO usar a estrutura squads/{name}/output/{run_id}/v{N}/ do OpenSquad (maquinaria desnecessaria).

`references/pipeline.md`: detalhamento das 5 fases, checkpoints obrigatorios, ordem de recrutamento dos agents, e a tabela de injecao formato->best-practice. Header `# SOURCE`, zero `<!--`.

Incluir secao "Quando NAO usar /marketing": codigo/feature (isso e GSD), deploy (Lovable), design de UI de app (ui-ux-pro-max direto).
