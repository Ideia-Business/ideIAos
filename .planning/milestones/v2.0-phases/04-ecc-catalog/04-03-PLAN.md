## Tasks

### Task 1 — Skills de teste e migração (tdd, e2e-testing, database-migrations)

Criar `security/quarantine/04-03/<name>/SKILL.md`. Cada SKILL.md: frontmatter + header MIT + corpo (~50-90 linhas). Estrutura padrão do corpo: **Quando usar / Quando NÃO usar / Processo / Output / Anti-patterns**.

**`tdd`** — frontmatter `name: tdd`, description: "Test-Driven Development: RED→GREEN→REFACTOR. Use proactively para lógica de negócio com I/O definido (validação, transformação, regra, endpoint). Escreve teste que falha ANTES da implementação." Corpo:
- Quando usar: pode-se escrever `expect(fn(input)).toBe(output)` antes de `fn`.
- Quando NÃO: UI/estilo, glue code, config.
- Processo: RED (teste falha) → GREEN (mínimo p/ passar) → REFACTOR (limpar, testes verdes). 1 commit por fase.
- Output: arquivo de teste + implementação + 2-3 commits atômicos.
- Anti-pattern: escrever implementação antes do teste; testar o mock.

**`e2e-testing`** — `name: e2e-testing`, description: "Testes end-to-end de fluxos críticos (Playwright/Cypress). Use proactively para fluxos de usuário que NÃO podem quebrar (login, checkout, deploy)." Corpo:
- Quando usar: fluxo crítico multi-tela; regressão de produção.
- Quando NÃO: lógica isolada (use tdd/unit).
- Processo: identificar o caminho feliz crítico → escrever teste do fluxo real (não mock) → rodar headless em CI → estabilizar flakiness (waits explícitos, não sleep).
- Output: spec e2e + comando de execução + passando em CI.
- Anti-pattern: e2e para tudo (lento/frágil); sleeps fixos.

**`database-migrations`** — `name: database-migrations`, description: "Workflow seguro de migrations (Supabase/SQL): diff → preview branch → aplicar → rollback plan. Use proactively antes de qualquer mudança de schema." Corpo:
- Quando usar: DDL, nova tabela/coluna/policy.
- Quando NÃO: query sem mudança de schema.
- Processo: `supabase db diff` → testar em preview branch → checklist RLS (delegar ao agent rls-reviewer) → aplicar → registrar rollback.
- Gotchas (do vault): nunca `DROP COLUMN` sem fallback; `service_role` não respeita RLS; testar com dump sanitizado.
- Output: migration versionada + plano de rollback.

### Task 2 — Skills de exploração e research (deep-research, codebase-onboarding, code-tour)

**`deep-research`** — `name: deep-research`, description: "Pesquisa profunda multi-fonte com retrieval iterativo: orquestrador avalia retorno, faz follow-up, máx 3 ciclos. Use proactively para decisões técnicas com trade-offs ou domínio desconhecido." Corpo:
- Quando usar: escolher entre opções; domínio novo; decisão arquitetural.
- Quando NÃO: resposta já conhecida (não gaste ciclos).
- Processo (iterative retrieval — regra orchestration.md): definir objetivo + query → buscar → avaliar lacuna → follow-up (máx 3 ciclos) → sintetizar em arquivo.
- Output: documento de research com fontes, opções, recomendação.
- Anti-pattern: 1 busca rasa; ciclos infinitos sem convergir.

**`codebase-onboarding`** — `name: codebase-onboarding`, description: "Mapeia uma codebase desconhecida em um guia navegável: arquitetura, entrypoints, convenções, gotchas. Use proactively ao entrar em projeto/módulo novo." Corpo:
- Quando usar: primeiro contato com repo/módulo.
- Processo: detectar stack → mapear entrypoints → tracejar 1 fluxo end-to-end → listar convenções e gotchas. (Pode delegar busca ao agent code-explorer em haiku.)
- Output: ONBOARDING.md curto: arquitetura, como rodar, onde mexer, armadilhas.

**`code-tour`** — `name: code-tour`, description: "Gera um tour guiado de uma feature/fluxo específico — sequência comentada arquivo:linha → arquivo:linha. Use proactively para explicar 'como X funciona' a um humano ou IA." Corpo:
- Quando usar: explicar um fluxo específico (não a codebase toda).
- Processo: localizar entrypoint → seguir as chamadas → anotar cada salto com 1 linha de "o que acontece aqui".
- Output: tour numerado (passo → arquivo:linha → explicação).
- Diferença de codebase-onboarding: tour é de UM fluxo; onboarding é do projeto.

### Task 3 — Skills de design, qualidade e custo (api-design, accessibility, benchmark-optimization-loop, cost-tracking)

**`api-design`** — `name: api-design`, description: "Design de APIs (REST/RPC) com contratos claros: tipos de request/response, status codes, versionamento, erros consistentes. Use proactively antes de implementar endpoint novo." Corpo:
- Quando usar: endpoint/contrato novo.
- Processo: definir contrato (request/response types) ANTES da implementação → status codes corretos → erros consistentes (shape único) → versionamento → idempotência onde aplica.
- Output: contrato (tipos + tabela de endpoints) que o executor implementa. Pareia bem com TDD.
- Anti-pattern: implementar antes de definir contrato; erros ad-hoc.

**`accessibility`** — `name: accessibility`, description: "Acessibilidade web (WCAG 2.1 AA): semântica, contraste, teclado, ARIA, foco. Use proactively em qualquer UI nova ou antes de publicar. Complementa /web-quality (auditoria) e /ui-ux-pro-max." Corpo:
- Quando usar: UI nova/alterada; antes de deploy.
- Quando NÃO: backend puro.
- Processo: HTML semântico primeiro (ARIA só quando necessário) → contraste AA → navegação por teclado + foco visível → labels em inputs → testar com leitor de tela.
- Output: checklist WCAG marcado + correções aplicadas.
- Relação: `/web-quality` audita (mede), esta skill orienta a CONSTRUIR acessível.

**`benchmark-optimization-loop`** — `name: benchmark-optimization-loop`, description: "Loop medir→otimizar→medir para performance: estabelece baseline, muda uma coisa, re-mede, mantém só se melhorou. Use proactively antes de qualquer otimização. Pareia com agent performance-optimizer." Corpo:
- Quando usar: antes de otimizar qualquer coisa.
- Processo: definir métrica + baseline → 1 mudança → re-medir → comparar → manter/descartar → repetir.
- Output: tabela baseline → mudança → resultado, com decisão por linha.
- Anti-pattern: otimizar sem medir; mudar várias coisas de uma vez.

**`cost-tracking`** — `name: cost-tracking`, description: "Rastreia e reduz custo de tokens/modelo por ação: model routing (haiku/sonnet/opus), MCP→CLI, contexto enxuto, strategic compact. Use proactively em sessões longas ou ao escolher modelo de um agent." Corpo:
- Quando usar: sessão longa; definir `model:` de agent; gargalo de custo.
- Processo: classificar a ação (busca/repetitivo → haiku; review/impl → sonnet; arquitetura/segurança → opus) → preferir CLI+skill a MCP pesado → contexto enxuto → `/compact` estratégico (~50 tool calls).
- Output: recomendação de modelo por ação + estimativa de economia (haiku ~5x mais barato que opus).
- Relação: complementa `source/rules/common/token-economy.md`.
