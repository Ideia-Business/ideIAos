### Task 1 — Schema do instinct + doc do layout (CONTRATO PRIMEIRO)

Definir o contrato antes das skills que o produzem (interface-first). `docs/instincts/instincts-layout.md` (`mkdir -p docs/instincts` — já criado se 05-01 rodou; o `mkdir -p` é idempotente). Header `# SOURCE: IdeiaOS v2`. Conteúdo (~60 linhas):

- **Árvore:**
  ```
  ~/.ideiaos/
  └── instincts/
      ├── project/
      │   └── <projeto-slug>--<instinct-slug>.md
      └── global/
          └── <instinct-slug>.md
  ```
  scope=project fica em `project/` (prefixado pelo slug do projeto p/ evitar colisão); scope=global em `global/`.

- **Schema do instinct (frontmatter YAML + corpo)** — este é o CONTRATO que `/instinct-analyze`, `/learn` (esta plan) e `/evolve` (05-03) compartilham:
  ```markdown
  ---
  trigger: "ao editar arquivo .ts/.tsx"
  action: "rodar tsc --noEmit antes de assumir que compila"
  confidence: 0.6          # 0.3 (fraco) .. 0.9 (forte). Nunca 1.0.
  domain: "typescript"      # área: typescript, supabase, git, testing, ...
  scope: "project"          # project | global
  project: "ideiapartner"   # presente só se scope=project
  evidence_count: 3         # quantas observações/sessões sustentam
  created: "2026-06-11"
  updated: "2026-06-11"
  source: "instinct-analyze" # instinct-analyze | learn
  ---
  # SOURCE: IdeiaOS v2

  ## Evidência
  - <bullets abstratos: padrão observado, sem conteúdo de arquivo nem secrets>

  ## Falsos positivos
  - <quando NÃO aplicar, se conhecido>
  ```

- **Regras de confidence:** análise automática começa em 0.3–0.5; cada nova evidência do mesmo trigger soma ~0.1 (cap 0.9); `/learn` manual entra em 0.5; ≥0.7 = "maduro" (elegível a `/evolve`).
- **Dedup:** chave = `slug(trigger)`. Mesmo slug → incrementa `evidence_count`, recalcula `confidence`, atualiza `updated`. Nunca cria duplicata.
- **Decay (curadoria):** instinct não reforçado há muito tempo pode ter confidence reduzida na próxima análise (regra detalhada e aplicada por `/evolve` em 05-03; aqui só documentar o campo `updated` que habilita isso).
- **Sync multi-máquina:** 1 arquivo por instinct (não um índice monolítico) → merges de iCloud/autosync raramente colidem; quando colidem, é num arquivo só, fácil de resolver. `~/.ideiaos/instincts/` propaga pelo mesmo mecanismo de sync já existente. Não versionar em git de projeto.
- **Quem escreve / lê:** escrevem `/instinct-analyze` e `/learn`; leem `/instinct-status`, `/evolve`, e `recall-learnings` Passo 6 (05-03).

### Task 2 — Skill /instinct-analyze (agente haiku background) + /instinct-status

`source/skills/instinct-analyze/SKILL.md`. Frontmatter:
```yaml
---
name: instinct-analyze
description: Analisa as observações coletadas (~/.ideiaos/observations/<projeto>/observations.jsonl) e destila instincts atômicos com confidence. Roda como agente haiku em background (barato). Use proactively ao retomar um projeto, após uma sessão longa, ou quando o marcador session_end indicar acúmulo de observações não-analisadas. Atualiza ~/.ideiaos/instincts/.
---
```
Linha `# SOURCE: IdeiaOS v2` após o frontmatter. Corpo PT-BR (~80 linhas):

- **Quando rodar:** retomada de projeto; após sessão longa; quando `observations.jsonl` cresceu desde a última análise (comparar `updated` dos instincts vs `ts` das observações).
- **Modelo:** invocar como **agente background haiku** (Task tool com `model: haiku` se disponível, ou nota explícita "rode em haiku — tarefa repetitiva/barata"). Princípio ECC: worker repetitivo = haiku, economia ~5x vs opus.
- **Pipeline:**
  1. Localizar `~/.ideiaos/observations/<projeto-slug>/observations.jsonl` (derivar slug do `basename $PWD`). Se não existir → "sem observações ainda" e sair.
  2. Ler as linhas (parse defensivo; pular linhas malformadas). Agrupar por padrão: sequências `(tool, ext, ok)` recorrentes, `bash_verb` recorrentes, eventos de erro (`ok:false`) seguidos de correção.
  3. Para cada padrão recorrente (≥2 ocorrências), formular UM instinct atômico: `trigger` (condição observável) + `action` (o que fazer). Abstrair — nunca citar conteúdo/secret.
  4. `confidence` inicial: 2 evidências → 0.3; 3-4 → 0.5; 5+ → 0.6. Cap 0.9.
  5. `domain` inferido do `ext`/`bash_verb` (ts→typescript, sql→supabase, git→git, test→testing...).
  6. `scope`: default `project`; marcar `global` só se o padrão é stack-agnóstico (mesma regra do extract-learnings `[global]`).
  7. **Dedup:** para cada instinct, computar `slug(trigger)`; se já existe arquivo → incrementar `evidence_count`, recalcular `confidence` (+~0.1, cap 0.9), atualizar `updated`; senão criar novo. `mkdir -p ~/.ideiaos/instincts/<scope>/`.
  8. NUNCA exceder confidence 0.9 na análise automática (só `/evolve` promove além, e mesmo assim 0.9 é o teto de instinct).
- **Privacidade:** validar que nenhum instinct gerado contém valores literais que pareçam secrets/paths sensíveis (reusar a noção de memory-hygiene). Em dúvida, abstrair mais.
- **Saída compacta:** bloco final `🧬 Instincts atualizados: N novos, M reforçados (domínios: ...)`.
- **Anti-padrões:** não criar instinct de evento único; não logar conteúdo; não duplicar; não inventar padrões sem evidência na jsonl.

`source/skills/instinct-status/SKILL.md`. Frontmatter:
```yaml
---
name: instinct-status
description: Lista os instincts aprendidos em ~/.ideiaos/instincts/ com barras de confidence, agrupados por domínio e scope (project/global). Use quando o usuário perguntar "o que você já aprendeu?", "quais instincts temos?", ou quiser revisar antes de /evolve.
---
```
Linha `# SOURCE: IdeiaOS v2`. Corpo PT-BR (~50 linhas):
- Varrer `~/.ideiaos/instincts/{project,global}/*.md`, ler frontmatter de cada (parse via leitura simples — pode usar `python3` inline numa Bash, sem jq).
- Render: agrupar por `scope` → por `domain`; cada instinct uma linha com **barra de confidence** (ex.: `[██████░░░░] 0.6`), `trigger → action`, `evidence_count`, `updated`.
- Marcar visualmente os **maduros (≥0.7)** como elegíveis a `/evolve`.
- Se `~/.ideiaos/instincts/` não existir/vazio → "Nenhum instinct ainda. Rode /instinct-analyze após algumas sessões."
- Saída ordenada por confidence desc dentro de cada domínio.

### Task 3 — Skill /learn (extração manual mid-session) + commit

`source/skills/learn/SKILL.md`. Frontmatter:
```yaml
---
name: learn
description: Extração manual mid-session — invoque na hora em que você acabou de resolver algo não-trivial e quer registrar a lição como instinct imediato (confidence 0.5), sem esperar a análise automática. Escreve direto em ~/.ideiaos/instincts/.
---
```
Linha `# SOURCE: IdeiaOS v2`. Corpo PT-BR (~50 linhas):
- **Quando:** o usuário acabou de descobrir/resolver algo que vale virar instinct AGORA (a análise automática só rodaria depois).
- **Gate leve:** 1 pergunta — "isso se repete em sessões futuras?" Se não, abortar com 1 linha (`📚 Nada a registrar — específico demais desta sessão`).
- **Processo:**
  1. Formular `trigger` + `action` atômicos (abstratos, sem secrets/conteúdo).
  2. `confidence: 0.5` (manual entra no meio — mais forte que análise automática de 2 evidências, menos que maduro).
  3. `domain` e `scope` (default project) conforme o assunto. `source: learn`.
  4. **Dedup** (mesma regra da Task 1): se `slug(trigger)` já existe → reforçar (`evidence_count += 1`, subir confidence até 0.9, `updated`), em vez de duplicar.
  5. `mkdir -p ~/.ideiaos/instincts/<scope>/` e escrever o arquivo no schema da Task 1.
- **Saída:** `🧬 Instinct registrado: "<trigger>" → "<action>" (confidence 0.5, domain <X>, scope <Y>)`.
- **Anti-padrões:** não registrar trivialidade; não duplicar; não inventar confidence alta (manual é sempre 0.5 no nascimento).
- **Relação:** `/learn` (manual, na hora) complementa `/instinct-analyze` (automático, em lote). Os dois escrevem no MESMO layout.
