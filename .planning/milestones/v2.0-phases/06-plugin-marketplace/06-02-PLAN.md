## Notes

- **Ordem de commit (crítico para o pre-commit hook):** A remoção de componentes (skills/agents/hooks/templates) dispara o pre-commit README-sync hook, que exige README atualizado no MESMO commit. Como a atualização do README (árvore + seção plugin) é escopo de 06-03, a execução real deve seguir **06-02 (deixa working tree pronto, sem commit) → 06-03 (atualiza README + roda check-readme-sync) → UM commit conjunto** cobrindo remoção + README, passando no hook SEM `--no-verify`. Alternativamente, se o executor preferir commits separados, 06-02 pode incluir o ajuste MÍNIMO da árvore do README — mas a decisão travada aqui é commit conjunto com 06-03 para manter o README coerente num só passo.
- **Por que depende de 06-01:** ordem de segurança — o repo nunca fica sem caminho de instalação. 06-01 cria o caminho plugin; 06-02 remove os fallbacks mortos. A remoção em si não usa nada de 06-01.
- **Hooks test-*:** `test-hooks.sh`, `test-typecheck-on-edit.sh`, `test-observe-hooks.sh` vivem em `source/hooks/` e NÃO são componentes de produto. A nova check-readme-sync.sh os exclui da auditoria (case test-*) → 06-03 não precisa listá-los no README (eram listados antes porque o loop antigo varria hooks/*.sh; agora ficam fora).
- **Recuperação:** tudo via `git rm` — `git checkout <commit-anterior> -- skills/` recupera se necessário. Histórico preservado.
- **Autosync:** pausado no Passo 0 da Task 1; o 06-03 religa (`launchctl start <label>`) DEPOIS do commit conjunto. Se 06-03 falhar, religar o autosync só após resolver o working tree (commit ou reset).
