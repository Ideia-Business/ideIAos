<task type="auto">
  <name>Task B1 (Wave 1, depends B0): assinar→verificar fail-closed (R-WP1 + R-WP2)</name>
  <files>source/agentd/verify-payload.sh</files>
  <action>
    `verify-payload.sh` verifica uma assinatura contra a lista pinada LOCAL. **Fixture de 2 pares**
    (cross-identity, mesma máquina): peerA (emissor) e peerB (verificador), com o pin de peerA no store
    local de peerB — peerB verifica a assinatura de peerA contra o PIN DE peerA (não contra a própria
    chave). O `role` autorizador vem da CHAVE PINADA casada — `role` de payload divergente é RECUSADO
    (R-WP2). Assinatura inválida/chave-não-pinada → recusa (R-WP1). sha256 nunca autoriza (nem
    sha256-correto-com-assinatura-ruim).
  </action>
  <acceptance_criteria>
    - VÁLIDO PASSA (cross-identity): peerA assina, peerB (com peerA pinado) verifica → exit 0.
    - PAYLOAD-TAMPER RECUSA (veneno canônico, imune ao quirk de armadura do ssh-keygen): virar 1 byte do PAYLOAD → exit != 0. NÃO usar "byte após o footer da assinatura" — o parser ssh-keygen ignora trailing bytes e ACEITARIA (verificado).
    - CHAVE NÃO-PINADA RECUSA: peerB' efêmero não-pinado assina → exit != 0.
    - PAPEL FORJADO RECUSA (R-WP2): payload `role:cto` mas peerA pinado como `role:dev` → exit != 0 (papel vem do pin).
    - SÓ-SHA256 RECUSA: payload só com sha256 (sem assinatura) → exit != 0.
    - DOWNGRADE RECUSA (sha256 não salva assinatura ruim): payload com sha256 CORRETO + assinatura inválida/não-pinada → exit != 0.
  </acceptance_criteria>
  <done>verify fail-closed: válido passa; inválido/não-pinado/papel-forjado/só-sha256 recusam.</done>
</task>
