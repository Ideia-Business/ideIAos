## Summary

- **The SPA read path is already decided and shipped in v14.0 — v14.1 EXTENDS it, does not re-choose.** v14.0 chose a **tiny loopback HTTP read server** (`apps/cockpit/server/read.js`, bound explicitly to `127.0.0.1:3073`) using Node's **built-in `node:sqlite` (`DatabaseSync`)** — NOT `better-sqlite3`, NOT sql.js/wasm, NOT a build-time fixture. The SPA (Vite dev on fixed `127.0.0.1:5273`) fetches `GET /machines` with a CORS origin allowlist. v14.1 adds endpoints (`/overview`, `/fleet`, `/vault`, `/verify`) to this same server and screens to the SPA. **CONTEXT.md says "better-sqlite3" — that is wrong; the code uses `node:sqlite`. Use `node:sqlite`.** (Landmine #1.)

- **There is no command channel yet — the agentd is `--once` collect-only.** `source/agentd/agentd.js` only does read-only collection and exits. For ⌘K, the minimal mechanism is to **add a second loopback HTTP endpoint (`POST /command`) to the existing `read.js` server** (or a sibling `command.js` on `127.0.0.1`), gated by a **fixed typed verb enum** (B1–B6, switch/case, never `exec(userInput)`), default-deny. This reuses the loopback-bind discipline already proven. A Unix domain socket is a viable alternative but adds zero security over a `127.0.0.1`-bound port here and costs browser-fetch ergonomics — recommend loopback HTTP with a typed enum.

- **The Flight Recorder reader contract is REAL and verified against this repo.** `git log -- versions.lock` returns exactly **13 commits**; the `gsd=` pin flip-flop is confirmed by `git show <H>:versions.lock | grep '^gsd='`. **Landmine #2:** the actual sequence has MORE nodes than doc 71's tidy `1.36.0→1.1.0→1.36.0→1.1.0` — and the reversals are driven by `wip: autosync` / `.local` commits (the Mac-mini daemon re-pinning `1.36.0`) overwriting human `1.1.0` commits. The actor classifier ALREADY EXISTS in `ingest.js:classifyActor()` — reuse it verbatim; do not re-invent.

- **The two R14-06 gates and the recorder gate are NEW npm scripts that do not exist yet** (`test:zeroleak`, `test:recorder` are absent from both package.jsons). But their building blocks exist: `zeroleak-snapshot.sh` (the regex scanner, **dogfood-veneno already passes — exit 1 on planted `sk-ant-FAKEKEY`, verified**), `ttt-baseline.sh` + `ttt-median.sh` (terminal baseline N≥5 already recorded). **Landmine #3:** doc 78 specifies a TWO-layer detector (regex + Shannon entropy with name-based allowlist) but v14.0's `zeroleak-snapshot.sh` is **regex-only** (header note A-01 admits this). v14.1 must add the entropy layer OR explicitly defer it and document the gap.

- **Primary recommendation:** Build v14.1 as vertical slices on the v14.0 spine — extend `read.js` (read endpoints + `POST /command` enum + `/verify`), add the SPA screens (Overview/Frota/Cofre + ⌘K + Flight Recorder band) consuming those endpoints, and wire the three exit-code gates (`test:zeroleak` extending the existing scanner to 7 surfaces + dogfood, `test:recorder` re-deriving the tape from git, `ttt-median.sh --mode=bridge` for the <10s gate). Reuse `node:sqlite`, the loopback bind, the actor classifier, the black-gold tokens in `src/index.css`. Add only `recharts` (and only if a chart is actually rendered — the Overview sparkline can be hand-rolled SVG per doc 50 §10.4, avoiding the dep).

**Primary recommendation (one-liner):** Extend `apps/cockpit/server/read.js` (read endpoints + a typed `POST /command` enum + `/verify`) and add SPA screens consuming it over loopback; wire `test:zeroleak`/`test:recorder`/`ttt-median.sh --mode=bridge` as exit-code gates — reusing `node:sqlite`, the 127.0.0.1 bind, `ingest.js:classifyActor`, and `src/index.css` tokens verbatim.

### Pattern 1: Loopback bind, never bind-all (M-01)
**What:** every local server binds `127.0.0.1` explicitly; bind-all (`0.0.0.0`/`::`) is a gate failure.
**When:** all of `read.js` and any command endpoint.
**Example (from shipped code):**
```javascript
// Source: apps/cockpit/server/read.js:33-34,128 (verified on disk)
const HOST = '127.0.0.1';
server.listen(PORT, HOST, () => { /* ... */ });
```
Gate (already in 14.0-07 plan, reuse): `grep -Eq "host:\s*['\"]127\.0\.0\.1|listen\([0-9]+,\s*['\"]127\.0\.0\.1" read.js` AND `! grep -Eq "0\.0\.0\.0|host:\s*['\"]::" read.js`.

### Pattern 2: Typed verb enum, never arbitrary exec (OWASP LLM06 Excessive Agency)
**What:** ⌘K commands map a fixed string enum → a pre-written CLI invocation; the request body's verb is validated against the enum BEFORE any exec; arguments are constant (not user-supplied).
**When:** the `POST /command` handler.
**Example (recommended shape):**
```javascript
// command verbs are a CLOSED SET; default-deny. No string interpolation of user input into a shell.
const VERBS = {
  pause_autosync:   { cmd: ['scripts/autosync-pause.sh', 'on', 'via Cockpit'], arm: true },   // B1
  resume_autosync:  { cmd: ['scripts/autosync-pause.sh', 'off'],               arm: false },  // B2
  reseal_security:  { cmd: ['scripts/check-security-freshness.sh','--record','PASS','@security-reviewer'], arm: true }, // B3
  force_sync:       { cmd: ['launchctl','kickstart','-k',`gui/${process.getuid?.()}/com.ideiaos.gitautosync`], arm: false }, // B4
  kickstart_daemon: { cmd: null, arm: false }, // B5 — daemon name validated against a 2nd closed set {envsync,refresh-ai-security,cockpit}
  run_doctor:       { cmd: ['bash','scripts/idea-doctor.sh'], arm: false },     // B6 (read-only)
};
// handler: const v = VERBS[body.verb]; if (!v) return res.writeHead(403)... ; spawnSync(v.cmd[0], v.cmd.slice(1))
```
**Critical:** use `spawnSync(file, argsArray)` (no shell) — never `execSync(stringWithUserInput)`. B5's daemon name is the ONLY variable input and is validated against `{envsync, refresh-ai-security, cockpit}` (note: `cockpit`, NOT `missioncontrol` — doc 77 B5 still shows the stale `missioncontrol`; correct it).

### Pattern 3: Positive structural proof beats negative scan (Zero-Leak)
**What:** prove a secret CANNOT be present (schema has no `value` column) in addition to scanning for one.
**When:** the `test:zeroleak` gate.
**Example (from shipped schema):**
```sql
-- Source: source/console/schema.sql:58-67 (verified) — api_key has NO `value` column, by design
CREATE TABLE IF NOT EXISTS api_key ( project_slug TEXT, var_name TEXT, present INTEGER,
  expected INTEGER, risk_tier TEXT CHECK(...), file_mtime_epoch INTEGER, committed INTEGER, PRIMARY KEY(...) );
```
Gate: `sqlite3 db 'PRAGMA table_info(api_key)' | awk -F'|' '{print $2}' | grep -qix value && exit 1 || exit 0`.

### Pattern 4: Actor classification is deterministic string-match (reuse, do not re-invent)
**What:** classify a commit as autosync/daemon vs human by string rule.
**Example (from shipped code — reuse verbatim for Flight Recorder):**
```javascript
// Source: source/console/ingest.js:72-79 (verified)
function classifyActor(subject, author) {
  const subj = (subject || '').toLowerCase();
  const auth = (author  || '').toLowerCase();
  if (subj.startsWith('wip: autosync')) return 'autosync';
  if (auth.endsWith('.local'))          return 'autosync';
  if (auth.includes('[bot]@'))          return 'bot';
  return 'human';
}
```

### Anti-Patterns to Avoid
- **Reintroducing `mission-control`/`missioncontrol`** anywhere (B5 daemon name, ref, capability). Verified obsolete.
- **Using `better-sqlite3`** because CONTEXT.md said so — the code uses `node:sqlite`. Adding `better-sqlite3` would be a needless npm dep contradicting the shipped decision.
- **`execSync` with any user-derived string** in the command handler (Excessive Agency / RCE-equivalent).
- **Trusting doc 71's tidy 4-node flip-flop string** — the real tape has 13 nodes; render what `git show` returns, never a hand-typed sequence.
- **Animating remote-node freshness as continuous flow** — only the local heartbeat animates (spec "Frescor honesto").
- **Color as the only state signal** — always dot+text+icon (WCAG; doc 50 §10.5).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Open/query the read-model | a custom SQLite parser or a bundled DB engine | `node:sqlite` `DatabaseSync` (already used) | built-in, zero supply-chain, already the project standard |
| Loopback read endpoint | a new framework (express/fastify) | extend the existing `node:http` server in `read.js` | CONTEXT bans new deps; `node:http` already proven loopback-bound |
| git history replay | parse commit prose with an LLM | `git log --format` + `git show <H>:versions.lock | grep '^gsd='` | git IS the append-only event-store; deterministic, exit-code-verifiable; NL parsing is forbidden (`antifragile-gates`) |

**Key insight:** v14.0 deliberately minimized dependencies to keep the Zero-Leak surface auditable. Every "Don't Hand-Roll" here resolves to *reuse the v14.0 building block*, not *add a library* — the opposite of the usual greenfield advice, because the substrate already exists.

## Common Pitfalls

### Pitfall 1: `better-sqlite3` ghost dependency (Landmine #1)
**What goes wrong:** CONTEXT.md and the milestone PLAN say "SQLite via `better-sqlite3`". The actual code (`ingest.js:26`, `read.js:20`) uses Node built-in `node:sqlite`. A planner trusting the doc would add an unnecessary npm dependency (build surface + supply-chain risk) and diverge from the shipped pattern.
**Why:** the doc was written before/independently of the code; the 14.0-07 plan threat-model T-14.0-07-SC explicitly chose `node:sqlite` for "supply-chain surface nula".
**How to avoid:** use `node:sqlite`. Treat the CONTEXT "better-sqlite3" string as an `[ASSUMED]` that the code overrides.
**Warning signs:** `npm install better-sqlite3` appears in any plan task.

### Pitfall 3: Zero-Leak detector is regex-only, doc 78 requires 2 layers (Landmine #3)
**What goes wrong:** doc 78 §1.2 mandates a TWO-layer detector: (a) literal key regex + (b) Shannon entropy ≥4.0 with a name/shape allowlist. The shipped `zeroleak-snapshot.sh` is **regex-only** — its header explicitly notes "NOTA (A-01, escopo v14.0): detector é REGEX-only (sem camada de entropia/allowlist)". The 5 regex patterns (`sk-`, `gho_`, JWT two-segment, `service_role` value-shape, hex≥41) catch the known fixtures and the dogfood-veneno (verified: planted `sk-ant-FAKEKEY` → exit 1), but a NOVEL high-entropy key the regex doesn't know would slip through.
**Why:** v14.0 scoped the entropy layer out (A-01). v14.1's R14-06 elevates Zero-Leak to the P0 release gate over 7 surfaces.
**How to avoid:** the planner must DECIDE explicitly — either (a) add the entropy layer in v14.1 (with the name-based allowlist: `machine_id`=sha256[:12], commit SHAs, `supabase_project_id`, `input_hash`, audit-log hashes — allowlist by NAME/shape, never value), or (b) keep regex-only and document the residual gap in the SUMMARY. Given R14-06 calls Zero-Leak the P0 gate and the spec scenario "invariante de release" is unconditional, (a) is the safer reading. The dogfood-veneno alone does not prove the entropy layer (the veneno is a `sk-` key the regex catches) — add a SECOND veneno with a novel high-entropy non-`sk-` token to actually exercise the entropy layer if added.
**Warning signs:** `test:zeroleak` passes on a planted random 40-char base64 token that matches no known prefix.

### Pitfall 4: `ttt-median.sh` has no `--mode=bridge` flag yet (Landmine #4)
**What goes wrong:** doc 78 §2.2 specifies the gate `ttt-median.sh --mode=bridge | awk '$2>=10{f=1} END{exit f}'`. The shipped `ttt-median.sh` accepts an optional TSV path arg but **no `--mode` flag**; it computes the median over ALL non-placeholder rows (it filters OUT `bridge-placeholder`/`bridge-dry-run`, the opposite of selecting bridge rows). And `ttt-baseline.sh --mode=bridge` currently only writes a `bridge-placeholder` row (the real bridge measurement path does not exist).
**Why:** v14.0 built the terminal baseline + the structural stub for bridge; the actual bridge measurement is v14.1's job.
**How to avoid:** v14.1 must (1) extend `ttt-baseline.sh` to record REAL bridge measurements (mode=`bridge`, real seconds, not placeholder) and (2) extend `ttt-median.sh` to accept `--mode=bridge` selecting only `$2=="bridge"` rows. Keep the same TSV format `<journey>\t<mode>\t<seconds>\t<epoch>`.
**Warning signs:** the gate command errors or returns the terminal median (≈0.002s) instead of a real bridge median.

### Pitfall 5: git tests in the live repo (regression of a known learning)
**What goes wrong:** running Flight Recorder / Zero-Leak / ref tests against the live repo while autosync runs corrupts results (`autosync-races-ai-git-surgery`, `verify-guards-in-sandbox-not-live-repo`).
**Why:** the git-autosync daemon does `git add -A` + commit + push every 900s and can capture the test window; `git checkout` to set up a fixture can fail silently.
**How to avoid:** run git tests in a `/tmp` `git init` sandbox with autosync PAUSED (`autosync-pause.sh on`). **Verified: autosync is already PAUSED** on this machine (`autosync-pause.sh status` → `⏸ PAUSADO`). The Flight Recorder's READ of the live repo's `versions.lock` history is read-only (`git log`/`git show`) and safe; only WRITE-side ref tests need the sandbox.
**Warning signs:** `git status --porcelain` non-empty after a test; a `wip: autosync` commit appearing mid-test.
