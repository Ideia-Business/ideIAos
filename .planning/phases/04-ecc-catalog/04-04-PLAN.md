## Tasks

### Task 1 — 3 skills-receita via quarentena + /ideiaos-catalog

`mkdir -p security/quarantine/04-04`. Criar 3 skills-receita (derivadas dos guias ECC → quarentena) + a skill `/ideiaos-catalog` (própria IdeiaOS → header IdeiaOS, não precisa de quarentena mas pode passar no scan sem problema).

**`two-instance-kickoff`** (`name: two-instance-kickoff`) — description: "Kickoff de projeto novo com 2 instâncias em paralelo: uma faz scaffold/setup, outra faz research do domínio. Use proactively ao iniciar projeto do zero." Corpo (~50 linhas):
- Quando usar: projeto novo, do zero.
- Quando NÃO: projeto existente.
- Processo: Instância A = scaffold (setup, estrutura, deps); Instância B = deep-research do domínio (pareia com skill deep-research). Convergir num plano único.
- Princípio ECC: "minimum viable parallelization" — só 2 instâncias, com objetivo claro cada.
- Output: scaffold pronto + documento de research, prontos para o primeiro `/gsd-plan-phase`.

**`llms-txt`** (`name: llms-txt`) — description: "Gera docs LLM-otimizadas (llms.txt) para uma codebase/produto: índice navegável que um agente lê para entender o projeto rápido. Use proactively para projetos que serão consumidos por IA." Corpo (~45 linhas):
- Quando usar: expor projeto/produto para consumo por IA.
- Processo: mapear entrypoints e docs → gerar `llms.txt` (índice + links + 1 linha por seção) → manter enxuto (sinal, não ruído).
- Output: arquivo `llms.txt` na raiz do projeto-alvo.

**`mcp-to-cli`** (`name: mcp-to-cli`) — description: "Converte uma integração MCP pesada em skill + CLI equivalente, reduzindo tokens e tools ativos. Use proactively quando um MCP consome muito contexto ou há >10 MCPs ativos. Implementa a regra mcp-hygiene." Corpo (~55 linhas):
- Quando usar: MCP pesado/redundante; >10 MCPs ativos; >80 tools.
- Processo: identificar as operações REALMENTE usadas do MCP → mapear cada uma para um comando CLI → empacotar como skill com exemplos → desativar o MCP por projeto (`disabledMcpServers`).
- Exemplo: Supabase MCP → skill que chama `supabase` CLI (`db diff`, `db push`, `migration`).
- Relação: aplica `source/rules/common/mcp-hygiene.md` e `token-economy.md`.
- Output: skill CLI + MCP desativado + economia de tools/tokens documentada.

**`ideiaos-catalog`** (`name: ideiaos-catalog`) — skill PRÓPRIA IdeiaOS (header `# SOURCE: IdeiaOS v2` opcional; NÃO é ECC). description: "Lista os módulos do IdeiaOS (agents, skills, hooks, rules, templates) a partir de manifests/modules.json — instalados vs disponíveis, filtrável por kind e stack. Use quando o usuário pergunta 'o que tem disponível', 'quais agents/skills existem', ou quer instalar algo sob demanda." Corpo (~70 linhas):
- Quando usar: "o que tem disponível?", "quais skills/agents?", "instala X sob demanda".
- Processo:
  1. Ler `manifests/modules.json`.
  2. Para cada módulo, determinar **instalado** vs **disponível** (instalado = arquivo existe em `~/.claude/` ou `.cursor/` conforme target; disponível = está no manifesto mas não instalado).
  3. Filtrar por `--kind agent|skill|hook|rule|template` e por `--stack <stack>` (via `installStrategy: stack:<x>`).
  4. Apresentar tabela: id · kind · installStrategy · status (✅ instalado / ⬜ disponível).
- Comando de instalação sob demanda: orientar `bash scripts/build-adapters.sh` ou `bash setup.sh` conforme o módulo.
- Output: tabela do catálogo + sugestão de como instalar o que falta.
- Inclui bloco bash de referência:
  ```bash
  python3 -c "import json;d=json.load(open('manifests/modules.json'));[print(m['kind'],m['id'],m['installStrategy']) for m in d['modules']]"
  ```

Escrever as 3 receitas em `security/quarantine/04-04/<name>/SKILL.md`, scanear, promover. `ideiaos-catalog` pode ir direto para `source/skills/ideiaos-catalog/SKILL.md` (conteúdo próprio), mas rodar o scan nela também (boa higiene).

```bash
bash security/scan-absorbed.sh security/quarantine/04-04/
for d in security/quarantine/04-04/*/; do
  name="$(basename "$d")"; mkdir -p "source/skills/$name"; mv "$d/SKILL.md" "source/skills/$name/SKILL.md"
done
```
