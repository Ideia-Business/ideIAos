# Fase 12 — Plan 01: Runner LLM + Política de Saída

## Objetivo

`run-evals.sh` sai do modo "veredito manual" para execução real: `run_case_with_model()` chama `claude -p` com o prompt do caso, avalia a resposta contra os critérios via grep, e retorna `pass` ou `fail`. Flag `--ci` habilita a política de bloqueio: qualquer falha em caso `pass^k` termina com `exit 1`; falha em `pass@k` emite warning sem bloquear. Resultados gravados em `evals/results/`.

## Contexto

O script atual tem `run_case_with_model()` como stub marcado `TODO` que coleta veredito do operador via stdin. A seção Setup/Prompt e os Critérios de Aprovação já estão parseados no loop principal. A extensão deve ser **minimamente invasiva**: só a função `run_case_with_model()` muda, o loop principal e o parser de frontmatter ficam intactos.

Restrições de ambiente:
- Bash 3.2 compatível (macOS) e bash 5 (ubuntu CI) — sem `mapfile`, sem `declare -A` associativos com `=()` portável apenas bash 4+.
- Sem `jq` — output JSON manual com `printf`.
- O comando headless é `claude -p "<prompt>"` com `--no-color` e `--print` equivalente ao modo `-p`.
- `ANTHROPIC_API_KEY` deve estar no environment; se ausente e `--ci` ativo, `run_case_with_model()` retorna `skip` (não `fail`) e imprime aviso — nunca trava.
- Timeout por caso: 90 segundos via `timeout 90 claude -p ...`; se `timeout` não estiver disponível (macOS sem GNU coreutils), usar `perl -e 'alarm 90; exec @ARGV' --` como fallback.

## Tarefas

### Task 1 — R3-11: Implementar run_case_with_model() com execução real

**Arquivos:** `evals/run-evals.sh`

Substituir o corpo de `run_case_with_model()` preservando a assinatura `run_case_with_model(local _case_file="$1")`.

Lógica a implementar:

1. **Guard de API key:** se `ANTHROPIC_API_KEY` estiver vazio e `CI_MODE` for 1, imprimir `"  [AVISO] ANTHROPIC_API_KEY ausente — caso marcado como skip"` e retornar `"skip"`. Se `ANTHROPIC_API_KEY` vazio e modo interativo, cair no fallback de veredito manual existente (preservar comportamento original).

2. **Extrair prompt e critérios:** reutilizar `extract_section "$_case_file" "Setup/Prompt"` e `extract_section "$_case_file" "Critérios de Aprovação"`. Remover blocos de código fenced (linhas ``` ) e linhas de cabeçalho Markdown do prompt antes de enviar.

3. **Chamar claude headless:** Construir a chamada como:
   ```
   response=$(timeout 90 claude -p "$prompt_text" --no-color 2>/dev/null)
   ```
   Se `timeout` não estiver disponível no PATH, usar:
   ```
   response=$(perl -e 'alarm 90; exec @ARGV' -- claude -p "$prompt_text" --no-color 2>/dev/null)
   ```
   Detectar disponibilidade de `timeout` com `command -v timeout >/dev/null 2>&1`.
   Capturar exit code: se diferente de 0 e não for timeout (exit 124), retornar `"fail"` com log `"  [ERRO] claude saiu com $exit_code"`.

4. **Avaliar resposta:** Extrair linhas de critério (linhas que começam com `- [ ]`) dos critérios. Para cada critério, extrair o texto após `- [ ] `. Fazer grep case-insensitive da frase-chave no `$response`. Se qualquer critério falhar no grep, veredito = `"fail"`. Se todos passarem, veredito = `"pass"`. Critérios com negação (contêm "NÃO" ou "NUNCA" em maiúsculo) são invertidos: grep encontrando o padrão = `fail` (Claude fez o que não devia). Para detectar negação: `echo "$criterion" | grep -qE "\bNÃO\b|\bNUNCA\b"`.

5. **Gravar resultado JSON:** Ao final de cada caso, emitir uma linha JSON para `$RESULTS_FILE` (variável global definida antes do loop). Formato:
   ```json
   {"id":"EVAL-001","metric":"pass^k","severity":"🔴","verdict":"pass","ts":"2026-06-12T10:00:00Z"}
   ```
   Usar `date -u +"%Y-%m-%dT%H:%M:%SZ"` para o timestamp.

6. **Variável CI_MODE:** adicionar `CI_MODE="${CI_MODE:-0}"` nos defaults. Adicionar flag `--ci` no parse de argumentos: `--ci) CI_MODE=1; DRY_RUN=0; shift ;;`. Quando `CI_MODE=1`, sobreescrever `DRY_RUN=0` para garantir execução real.

7. **Política de saída no sumário** (modificar o bloco de sumário existente):
   Ao final, se `CI_MODE=1`:
   - Se `pass_hat_total > 0` e `pass_hat_aprovados < pass_hat_total`: `exit 1` com mensagem `"BLOQUEIO: $((pass_hat_total - pass_hat_aprovados)) invariante(s) pass^k reprovada(s)"`.
   - Se apenas falhas pass@k: `exit 0` com mensagem `"AVISO: $((pass_k_total - pass_k_aprovados)) capacidade(s) pass@k reprovada(s) — não bloqueia CI"`.

8. **Inicializar RESULTS_FILE** antes do loop: criar `evals/results/` se não existir; definir `RESULTS_FILE="${SCRIPT_DIR}/results/$(date +%Y%m%d-%H%M).jsonl"`. No modo `--dry-run`, não criar o arquivo.

**Verificação:**

- `bash evals/run-evals.sh --dry-run 2>&1 | grep -c "dry-run"` retorna 22
- `bash evals/run-evals.sh --list 2>&1 | grep -c "EVAL-"` retorna 22
- `CI_MODE=0 ANTHROPIC_API_KEY="" bash evals/run-evals.sh --dry-run` exit 0 (nunca trava sem key)

**Critério de conclusão:** `bash -n evals/run-evals.sh` exit 0; `--dry-run` continua funcionando exatamente como antes (22 casos listados, nenhum veredito pedido); `--ci` flag reconhecida (sem "ERRO: argumento desconhecido").
