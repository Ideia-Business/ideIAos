<objective>
Build the IdeiaOS eval suite: a methodology README, a canonical case format, ≥20 eval cases
seeded from REAL documented incidents (ideiapartner INC-3xx + bugs, nfideia learnings/postmortems,
IdeiaOS's own VERIFICATION deviations), and a runner skeleton.

Purpose: Per ECC absorption plan Fase 7, convert real product failures into regression evals.
pass@k measures "must work" (does it succeed in k attempts), pass^k measures consistency (does
it succeed in ALL k attempts). The suite is designed for incremental growth — adding a new real
incident later is a single new case file, no code change.

Output: evals/README.md, evals/_TEMPLATE.md, evals/cases/index.md, evals/cases/*.md (≥20),

`#!/usr/bin/env bash` then `# SOURCE: IdeiaOS v2`. NEVER use `<!--`.

**evals/README.md** — methodology (PT-BR, ~80-140 lines):
  - Linha 1: `# SOURCE: IdeiaOS v2`.
  - Propósito: suíte de regressão que converte incidentes reais dos produtos (ideiapartner,
    nfideia) e desvios do próprio IdeiaOS em casos de avaliação. Toda falha vivida vira um caso.
  - Metodologia pass@k vs pass^k (defina os dois com clareza):
      * pass@k — "PRECISA FUNCIONAR": o comportamento esperado ocorre em PELO MENOS 1 de k
        tentativas. Use para capacidades que só precisam ser alcançáveis.
      * pass^k — "CONSISTÊNCIA": o comportamento esperado ocorre em TODAS as k tentativas.
        Use para invariantes de segurança/financeiro onde 1 falha já é inaceitável (ex: nunca
        fazer INSERT cego em client_subscriptions; nunca vazar cache multi-tenant).
      * Tabela orientando: invariantes de segurança/dados financeiros → pass^k; capacidades de
        produtividade → pass@k. Defina k padrão (sugestão k=5) e como reportar (m/k aprovados).
  - Estrutura da suíte: evals/cases/*.md (um caso por arquivo), evals/cases/index.md (roster),
    evals/_TEMPLATE.md (formato canônico), evals/run-evals.sh (runner).
  - Como executar: explique que a execução do modelo é MANUAL/SEMI-AUTOMÁTICA por design —
    `run-evals.sh` apresenta o caso (prompt/setup + critérios) e registra o veredito do operador;
    não chama LLM nem precisa de API key. Caminho futuro: plugar execução automática quando houver
    harness/key (deixe o ponto de extensão nomeado no script).
  - **Integração gsd-verify-work** (ponto de integração — requisito F07-VERIFY-INTEGRATION):
    documente que `gsd-verify-work` (skill GSD, `~/.claude/skills/gsd-verify-work/`) roda UAT
    conversacional por fase e gera `{phase}-UAT.md`. O ponto de integração é: ao fechar uma fase,
    casos de eval relevantes àquela fase devem ser executados como parte do UAT, e qualquer NOVO
    incidente descoberto no UAT deve virar um caso novo em evals/cases/ (loop fechado: incidente →
    caso → regressão). Como gsd-verify-work é uma skill do framework (não componente source/ do
    IdeiaOS), a integração é por REFERÊNCIA documentada aqui — não editamos a skill. Inclua um
    snippet de "como referenciar a suíte no fechamento de fase".
  - Convenção de IDs de caso: `EVAL-NNN` sequencial; cada caso aponta a `source` (incidente real).
  - Crescimento incremental: adicionar um incidente = copiar _TEMPLATE.md → cases/EVAL-NNN-slug.md
    + uma linha no index.md. Nenhuma mudança de código.

**evals/_TEMPLATE.md** — formato canônico de um caso (campos obrigatórios, com placeholders):
  - Linha 1: `# SOURCE: IdeiaOS v2`.
  - Campos (em frontmatter YAML + corpo): `id` (EVAL-NNN), `title`, `source` (projeto + doc do
    incidente real), `mode` (dev|review|research — qual contexto exercita), `metric` (pass@k|pass^k),
    `k` (nº de tentativas), `severity` (🔴|🟡|⚪).
  - Seções do corpo: **Setup/Prompt** (o prompt exato dado ao Claude + qualquer estado inicial),
    **Comportamento esperado** (o que um bom resultado faz), **Critérios de aprovação** (checklist
    objetivo e verificável — 2 a 5 itens), **Anti-comportamento** (a falha original que este caso
    previne). Mantenha curto e executável.

**evals/run-evals.sh** — runner skeleton (PT-BR comments, ~80-130 lines):
  - Linha 1: `#!/usr/bin/env bash`. Linha 2: `# SOURCE: IdeiaOS v2`. `set -uo pipefail`.
  - Resolve o dir de casos relativo ao script (`$(cd "$(dirname "$0")" && pwd)/cases`).
  - Args: `--case ID` (roda só um caso, casa por prefixo EVAL-NNN no nome do arquivo),
    `--dry-run` (lista o que faria — nome, source, métrica — sem pedir veredito), `--list`
    (imprime o roster a partir dos arquivos), `--help`. Args desconhecidos → erro + uso.
  - Comportamento padrão (sem args): itera cases/*.md em ordem; para cada caso imprime
    id/title/mode/metric/k + o bloco "Setup/Prompt" e "Critérios de aprovação" (extraídos do
    arquivo via parsing simples de seções), e — quando NÃO --dry-run — pausa pedindo veredito do
    operador (pass/fail/skip) lido de stdin; ao final imprime um sumário m/k por métrica.
    Em ambiente não-interativo (stdin não-tty) NÃO bloqueie: trate como --dry-run automaticamente.
  - Ponto de extensão nomeado: uma função `run_case_with_model()` claramente marcada como
    "TODO: plugar execução automática (API/harness) aqui" — hoje retorna veredito manual.
  - NUNCA exija edição do script para rodar. `--dry-run` deve funcionar sem nenhum input.
  - `chmod +x evals/run-evals.sh`.
