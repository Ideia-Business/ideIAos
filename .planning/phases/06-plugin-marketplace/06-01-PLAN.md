**Goal:** Estabelecer o IdeiaOS como marketplace de plugins instalável via `/plugin marketplace add Ideia-Business/IdeiaOS` + `/plugin install`, com 3 sub-plugins por perfil. Esta plan é **puramente aditiva** — cria apenas arquivos NOVOS (`.claude-plugin/`, `plugins/`, `scripts/build-plugins.sh`) e estende `manifests/modules.json`. **Não remove nem edita** `setup.sh`, `README.md`, `skills/`, `agents/`, `hooks/`, `templates/` — isso é escopo da 06-02 (remoção) e 06-03 (README). Por isso roda sozinha na Wave 1, sem conflito de arquivo com as outras.

### Esquema OFICIAL de plugin/marketplace (confirmado em code.claude.com/docs — 2026-06-12)

Fontes: `code.claude.com/docs/en/plugins`, `/plugin-marketplaces`, `/plugins-reference`, `/hooks`.

**1. `.claude-plugin/marketplace.json`** (na RAIZ do repo). Campos obrigatórios: `name` (kebab-case, único), `owner` ({name}), `plugins` (array). Cada entrada de plugin: `name` + `source` obrigatórios; `description`, `version`, `author` opcionais. `source` relativo (`"./plugins/x"`) é resolvido a partir da raiz do marketplace. Múltiplos plugins no MESMO repo = listar todos no mesmo `plugins[]`.

```json
{
  "name": "ideiaos",
  "owner": { "name": "Ideia Business", "email": "gustavo@redeideia.com.br" },
  "plugins": [
    { "name": "ideiaos-core",         "source": "./plugins/ideiaos-core",         "description": "Núcleo IdeiaOS: orquestrador /idea, 15 agents, 11 hooks de qualidade/memória/observação, skills de workflow e aprendizado." },
    { "name": "ideiaos-design-suite", "source": "./plugins/ideiaos-design-suite", "description": "Suíte de Design: ui-ux-pro-max, design, design-system, ui-styling, brand, banner-design, slides, motion, frontend-visual-loop, web-quality." },
    { "name": "ideiaos-lovable",      "source": "./plugins/ideiaos-lovable",      "description": "Camada Lovable: skill /lovable-handoff + doutrina de deploy e templates de handoff." }
  ]
}
```

**2. `.claude-plugin/plugin.json`** (dentro de CADA dir de plugin). Obrigatório: `name`. Recomendado: `version`, `description`, `author`. Componentes são auto-descobertos por convenção relativa à raiz do plugin: `skills/<nome>/SKILL.md`, `commands/*.md`, `agents/*.md`, `hooks/hooks.json`. NÃO é preciso listar paths se eles estiverem nos diretórios convencionais.

```json
{
  "name": "ideiaos-core",
  "version": "2.0.0",
  "description": "Núcleo IdeiaOS — orquestração, agents, hooks e skills.",
  "author": { "name": "Ideia Business", "email": "gustavo@redeideia.com.br" },
  "homepage": "https://github.com/Ideia-Business/IdeiaOS",
  "license": "MIT"
}
```

**3. `hooks/hooks.json`** (registro de hooks do plugin). Formato = mesmo dos hooks de settings.json, mas paths usam `${CLAUDE_PLUGIN_ROOT}`:

```json
{
  "description": "Hooks de qualidade, memória e observação do IdeiaOS",
  "hooks": {
    "PostToolUse": [
      { "matcher": "Edit|Write", "hooks": [ { "type": "command", "command": "\"${CLAUDE_PLUGIN_ROOT}\"/hooks/typecheck-on-edit.sh", "timeout": 60, "async": true, "asyncRewake": true } ] }
    ]
  }
}
```

**4. Fluxo de instalação:** `/plugin marketplace add Ideia-Business/IdeiaOS` (resolve owner/repo no GitHub) → `/plugin install ideiaos-core@ideiaos`. O sufixo `@ideiaos` = nome do marketplace (campo `name` do marketplace.json).

### Mapeamento de membership (decisão desta plan)

Fonte de verdade da membership = novo campo `plugins: ["<plugin>"]` em cada módulo de `manifests/modules.json` + doc `manifests/plugin-membership.md`. O `build-plugins.sh` LÊ esse mapeamento para decidir o que copiar.

- **ideiaos-core** (todos exceto design+lovable): 15 agents (todos `source/agents/*.md`), 11 hooks não-test (`source/hooks/*.sh` exceto `test-*`), 23 skills core: `accessibility, api-design, benchmark-optimization-loop, code-tour, codebase-onboarding, cost-tracking, cursor-continuation, database-migrations, deep-research, e2e-testing, evolve, extract-learnings, idea, ideiaos-catalog, ideiaos-setup, instinct-analyze, instinct-status, learn, llms-txt, mcp-to-cli, recall-learnings, tdd, two-instance-kickoff`.
- **ideiaos-design-suite** (10 skills, todas com `references/`/`scripts/`/`data/` → cp -R): `banner-design, brand, design, design-system, ui-styling, ui-ux-pro-max, slides, motion, frontend-visual-loop, web-quality`.
- **ideiaos-lovable**: skill `lovable-handoff` + (como dados/referência do plugin) `source/rules/lovable/deployment-protocol.md` e os 5 `source/templates/lovable/*.tmpl`.

### Constraints travadas

- Docs novos (`plugin-membership.md`, e o header de `build-plugins.sh`): header `# SOURCE: IdeiaOS v2`. JSON não tem header (não há comentário em JSON).
- NUNCA `<!--` em nada que o scan-absorbed.sh audite. JSON e .sh estão OK.
- Plugins são GERADOS — `plugins/` listado no `.gitignore`? **NÃO** — plugins precisam estar versionados no repo para o `/plugin marketplace add owner/repo` resolver via GitHub. Decisão: `plugins/` é COMMITADO (gerado + versionado, como artefato de release). `.gitignore` recebe apenas `plugins/**/.DS_Store`. Documentar no header do build-plugins.sh que `plugins/` é gerado e não deve ser editado à mão (edite `source/` e rode o build).
