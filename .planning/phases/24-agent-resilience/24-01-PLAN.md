<objective>
Adicionar resiliência/retomada ao spawn background do instinct loop (R6-02), adaptando a ideia
"capture state before a fallible operation, resume idempotently" do AIOX Agent Immortality Protocol —
sem copiar o JS, traduzindo para bash 3.2 fail-silent.

Hoje `observe-session-end.sh` spawna `/instinct-analyze` via `nohup env IDEIAOS_INSTINCT_SPAWN=1 timeout 120 claude ...`.
Se a máquina/sessão morre depois do spawn mas antes do filho terminar, a sentinela `.last-analyzed`
já foi escrita (gate em frente ao spawn), então as observações daquela janela podem ficar sem análise
— mas não há nenhum registro de que o spawn morreu no meio. Não há retomada nem limpeza.

Esta fase grava um **breadcrumb** (state file) antes do spawn e o limpa no retorno normal. Um novo hook
`instinct-recover.sh` (SessionStart) detecta breadcrumbs órfãos (processo morto + idade > timeout) e,
exatamente uma vez, retoma o spawn OU limpa com segurança — sempre respeitando as 5 barreiras anti-runaway.

Purpose: a lição runaway provou que estado de spawn precisa ser observável e recuperável sem reabrir a porta do loop infinito.
Output: breadcrumb writer/cleaner em observe-session-end.sh; recovery hook idempotente; teste de crash+retomada.
</objective>

<design_notes>
## Breadcrumb (state file)

Local: `~/.ideiaos/instincts/.spawn-${PROJ}.state` (FORA do repo — vive em ~/.ideiaos, como sentinela e logs).
Um por projeto (um spawn por projeto por vez; serializado pelo cooldown 30min).

Formato (sem-jq, montado por python3 — chave=valor por linha, simples de reler em bash 3.2):
```
pid=<pid do processo claude filho>
started_at=<ISO 8601 sem tz, lexicograficamente comparável>
project=<slug>
status=running
log=<caminho do LOG_FILE daquele spawn>
```

## Ciclo de vida
1. `observe-session-end.sh`, DEPOIS de passar os 5 gates e ANTES do `nohup ... claude`, grava o breadcrumb
   com `status=running`, `pid` do filho (capturado via `$!` após o spawn), `started_at`, `log`.
2. No retorno normal do filho (mesma subshell, via `wait`), o breadcrumb é removido (`rm -f`). Se o spawn
   estourar o `timeout 120`, `wait` ainda retorna e o breadcrumb é removido — fechamento limpo.
3. Se a sessão morre antes do `wait`, o breadcrumb fica órfão com `status=running`.

## Recovery (instinct-recover.sh — SessionStart, fail-silent exit 0)
Para cada `~/.ideiaos/instincts/.spawn-*.state`:
- Reler `pid`, `started_at`, `project`, `status`, `log`.
- **Liveness:** `kill -0 <pid> 2>/dev/null` → vivo? PULAR (spawn ainda em andamento, jamais duplicar).
- **Idade:** se `now - started_at <= 120s` (mesmo timeout do spawn), PULAR (pode estar terminando; sem corrida).
- Órfão confirmado (morto + > timeout): tratar UMA vez:
  - **Reivindicar atomicamente:** renomear o breadcrumb para `.spawn-<proj>.state.claimed-<pid_recovery>`
    (mv é atômico no mesmo FS) ANTES de qualquer ação — garante "exactly once" mesmo com 2 SessionStart concorrentes.
  - **Decisão retoma-vs-limpa:** reusar EXATAMENTE o mesmo bloco de gates do observe-session-end:
    `command -v claude` + obs mais recente que sentinela + cooldown 30min. Se os gates passam → re-spawnar
    (mesmo `nohup env IDEIAOS_INSTINCT_SPAWN=1 timeout 120 claude -p /instinct-analyze`, regravando breadcrumb fresco).
    Se cooldown ainda ativo OU sem obs novas → apenas limpar (`rm -f` do .claimed). Em ambos os casos o órfão some.
- O próprio `instinct-recover.sh` honra `[ -n "$IDEIAOS_INSTINCT_SPAWN" ] && exit 0` no topo (sessões de análise
  não recuperam nada — barreira #1 preservada).

## Por que isto NÃO enfraquece as 5 barreiras anti-runaway
| Barreira existente | Como o recovery a respeita |
|---|---|
| 1. IDEIAOS_INSTINCT_SPAWN guard | recover.sh sai cedo se setado; re-spawn seta `=1` |
| 2. Sentinela .last-analyzed (escrita antes do spawn) | re-spawn só após o gate `TS_OBS > TS_LAST`, e reescreve a sentinela antes do novo spawn |
| 3. Cooldown 30min | recover.sh aplica o mesmo gate ELAPSED<1800 → senão NÃO re-spawna (só limpa) |
| 4. timeout 120s no claude | re-spawn usa o mesmo `timeout 120` |
| 5. command -v claude | recover.sh sai se claude ausente |
Barreira nova (anti-corrida): claim-by-rename + kill -0 garante no máximo um spawn vivo por projeto.

## Adaptação do AIOX (não copiar JS)
Do `agent-immortality.js` reusamos os conceitos: state-commit antes da operação falível (breadcrumb),
claim atômico (rename, análogo ao `claimNext` da ReincarnationQueue), e "exactly once" via status transition.
Descartamos: autópsia/diagnóstico de causa, EventEmitter, locks com retry, fila multi-item — overkill para
um spawn-por-projeto serializado por cooldown. bash 3.2, sem-jq, sem dependências.
</design_notes>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sessão Claude → spawn background | hook dispara processo `claude` desacoplado; estado cruza via breadcrumb em disco |
| SessionStart concorrentes → mesmo breadcrumb | duas sessões podem ler o mesmo .spawn-<proj>.state simultaneamente |
| breadcrumb em disco → recovery | conteúdo do state file é entrada para decisão de re-spawn |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-01 | Elevation/DoS | recovery re-spawnando em loop (runaway redux) | mitigate | recovery aplica os 5 gates idênticos (sentinela+cooldown 1800s+IDEIAOS_INSTINCT_SPAWN+timeout 120+command -v claude); breadcrumb removido após tratamento |
| T-24-02 | Tampering/Race | dois SessionStart tratando o mesmo órfão → spawn duplo | mitigate | claim por `mv` atômico (.claimed-$$) antes de qualquer ação; perdedor do rename faz `continue` |
| T-24-03 | DoS | retomar um spawn cujo processo ainda está vivo | mitigate | gate `kill -0 <pid>` + gate idade > timeout antes de reivindicar |
| T-24-04 | Tampering | breadcrumb corrompido/parcial (crash durante a escrita) | mitigate | parse defensivo (campos ausentes → tratar como órfão e limpar); fail-silent, nunca aborta |
| T-24-05 | Info Disclosure | breadcrumb vazando paths/secrets | accept | só pid/timestamp/slug/log path em ~/.ideiaos (fora do repo, mesma sensibilidade da sentinela existente) |
| T-24-SC | Tampering | instalação de pacotes npm/pip/cargo | mitigate | nenhum pacote instalado nesta fase (bash + python3 stdlib); N/A |
